SDK and Sample Application Delivery

The Android SDK is shared as an .aar file and the sample application as a build-able source code to generate the required .apk file. This file can be integrated in Android Studio via Gradle file. Please note that the .aar file needs to be integrated into Lib (Library) folder, then in the project Build.gradle file.

To Download the Android SDK and Sample app Click here.

Sample Application

The sample application has two modes of operation:

  • Simulation mode (for more info about simulation Click here. )
  • Live mode
In simulation mode, the application runs a pre-configured scenario with pre-determined list of positions. This mode is intended for development phase. In live mode, the position is fetched from the position engine service and of course requires complete BLE beacon roll-out and site calibration data to have been completed. In both modes, connectivity to the position engine service and the presence of at least one BLE beacon is required.

The steps below explain how to run the sample application.

  • Install Sample App. You may need to build the sample app code to generate the needed .apk file. Grant all permissions manually as shown in the screen capture below:

  • Run the sample app. Click on Load Data button. This fetches all required data from the service over APIs:

  • To further ensure that the data has indeed been loaded successfully, click on the DB button and review that data which has been loaded to the SQLite tables.

  • After Loading data successfully, click on Login button. No username or password are required at this stage

  • After logging in, the map appears with the blue dot that shows the current user location. This is blue dot is based on a simulated position. A single hardcoded POI has been defined to demonstrate the navigation feature (shown as a grey square highlighted in red square). Simply click on the highlighted POI zone on the map and click on start navigation.



    Now that we have experienced the features of the sample app, we will now explain its inner workings.

    The sample app is intended to show a full wayfinding journey, present the two wayfinding modes available and showcase common API usage.

    PenNav has two wayfinding modes, namely: free-style and navigation. In free-style the user does not specify a destination. The position returned from the SDK reflects the current position of the user. In navigation mode, the user specifies a destination POI. The API offers a shortest path route from the current position to the POI entrance location. This route is returned as a list of edges. In this mode, an up-to-date position along the navigation path is returned by the SDK. The map image, the route and the location along the route must be drawn by the app developer according to the intended UI rendering guidelines. If the user deviates from the pre-calculated route, the SDK returns a wrong-path error. The app developer may elect to reflect that  in the app space for better navigation experience.

    In a multi-floor and/or multi-venue navigation scenario, the SDK continuously returns the venue (i.e. building) and the floor which shall be used by the app developer to update the currently displayed map image.

    The app developer may also call the API to list all available POIs to create – for example – a directory listing inside the app. Furthermore, the app developer shall use the API to retrieve the map image for a particular floor. They can also use the API to return the zone (defined as a polygon) which defines the boundaries of a particular POI on the map image. This zone definition can be leveraged in creating gesture interactions with the map image. This feature is implementation specific.

    The above is only a brief listing of the capabilities of the API. For a comprehensive list, the app-developer is referred to the API documentation.

    Within the sample app, we offer a hardcoded sample POI. It can be changed in the sample app code, by opening the com.penguinin_sampleData.Model.PoI\POI \Constructor, and modifying the following parameters:

    • ID.
    • Floor ID.
    • Name.
    • Description
    • Zone Points.

    However, the POI must exist in the list of available POIs for it to be reflected correctly in the sample app. The snippet below shows the POI description:

SDK Configuration

It is likely that during the development cycle, parameters for the SDK and the sample app will change. To change the SDK configuration from SDK folder open the com.penguinin_sampleUtil/SDKParameters, and you can start modifying the following parameters:

  • Data Server IP.
  • Data Service Name.
  • Position Server IP.
  • Position Service Name.
  • Fix Venue Value.
  • Use Https Connections Flag:
    • True: Https.
    • False: Http.
  • Simulation Mode flag:
    • True: Simulation scenarios run.
    • False: Live BLE scan run.

In this section we present the methods availed to the application developer through our SDK followed by best practices to integrate it within a mobile application.


4.1 Initialize

Returns initialization success or failure as Boolean value.

Operation

These steps describe the process of initialization:

  • Validate OS support and server connectivity.
  • Initialize input sources for positioning platform.
  • Detect current venue and get venue data (will use data API):
    • Venue specific positioning information
    • Floors details
    • Edges details
  • Perform device specific calibration
  • Establish connectivity requirements.
  • Save the results and Log for analytics (if supported)

Parameters

Parameters Description Value Type
UserID Front End UserID String
IPList Penguin Data API IP address (mandatory)
Penguin Position API IP address (mandatory)
Third party API IP address (optional) 		List(String)
context App main context Context
activity App main activity Activity
UseNoneIMU Enable Just JAR None IMU calculation Boolean

Response

Value Description Value Type
Result Success: 1 or Fail: 0
Error Code.
Warning Code.
separated by “_”
ex: “1_0_0” or “0_1_2” 		String

4.7 SetMode

Set the position mode and scanning time frequency:

  • Background:
    • Low frequency position scanning for logging and analytics.
  • Freestyle:
    • High frequency position scanning for trusted position on the venue. This mode assumes that the user is not wayfinding between a known source and a destination POI.
  • Navigation:
    • High frequency position scanning for trusted position while navigating to a known destination with a know route already identified.

Parameters

Parameters Description Value Type
ModeID BackGround, FreeStyle, Navigation Int
TimePeriod Time Period in ms Int

Response

Value Description Value Type
Success Updated the mode successfully Boolean

Modes

ID Description Scan Time (milliseconds)
1 Background (Default) 10000
2 Free Style 2000
3 Navigation 2000


4.8 GetMyPosition

Returns last detected position concatenated together with "_" separator:

  • Current Venue ID.
  • Current Floor ID.
  • Position point (X, Y).
  • Accuracy circle radius in pixels.
  • Current Edge ID.

Recommendation:

This method provides an update on the position of the mobile application user. It is not recommended to call it with high frequency. The recommended value is 500msec-2000msec and should be adjusted according to the desired UI behavior.

Operation

  • Concatenate VenueID, FloorID, X, Y, Accuracy Circle Radius , Current Edge ID together with "_" separator:
    • Ex:
      • Current Venue ID: 1.
      • Current Floor ID: 2.
      • My Position (X, Y): (3000, 5100).
      • Accuracy radius in pixels: 120.
      • Current Edge ID: 150.
      • Result: 2_3000_5100_120_150.

Parameters

N/A

Response

Value Description Value Type
Position VenueID, FloorID, X, Y, Accuracy Circle Radius , Current Edge ID
Concatenated together with "_" separator 		String
